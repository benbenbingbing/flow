## 自动化验证

- `npm run test:unit`：校验计算表达式引擎。
- `npm run test:integration`：校验动态表单/列表运行时配置解析、字段显示、按钮权限。
- `npm run test:functional`：校验系统核心功能模块，包括自定义列表/表单注册、列表工具栏/行操作注册、按钮组件注册、表单初始化器注册、联动显隐/禁用/必填/计算、流程节点配置、实体字段类型、API 功能方法清单、主要页面操作入口。
- `npm run test:page-config`：校验路由页面存在、核心页面标题、动态路由清单、动态运行时文件、字段组件注册覆盖。
- `npm run test:ui-config`：基于 Vue 编译器审计 68 个页面/组件中的 1235 个 Element Plus 配置控件，覆盖表单项、输入控件、表格列、弹窗、按钮、标签页等关键配置。
- `npm run test:e2e:mock`：构建生产包后启动 Vite Preview + Google Chrome Headless，通过 CDP mock 后端接口逐路由加载 23 个页面，校验页面可渲染、无浏览器运行时错误，并断言每个页面的关键显示文案；同时对 10 个安全功能入口执行真实点击交互并断言弹窗/配置区域显示。
- `npm run test:visual:real`：连接本地真实前端 `http://localhost:3000` 和真实后端 `http://localhost:8080`，使用系统 Google Chrome Headless 登录 `admin/admin` 后逐页截图验收 15 个核心页面，断言关键文案存在且无 Element Plus 错误消息。
- `npm run test:workflow:real`：使用真实 MySQL、Flowable 和后端接口，验证流程发布、节点解析、实体发布绑定、业务数据发起、待办生成、审批完成、进度和历史。
- `npm run test:workflow:config-real`：每次创建全新的两级审批流程、动态实体和两个审批账号，配置状态字典及两条节点状态映射，分别登录两个账号完成审批，并断言中间状态、最终自定义状态、待办归属和历史意见全部按配置生效。
- `npm run build`：校验所有页面和动态导入可以生产构建。

## 真实流程配置闭环

- 最终通过证据：`docs/workflow-closure/config-closure-26071504050vk.json`。
- 流程结构：管理员发起 -> 一级审批账号 -> 二级审批账号 -> 结束。
- 发布态校验：两个用户任务的候选人配置均被解析并保存到发布节点配置。
- 实体绑定校验：实体发布快照绑定到流程定义，`startProcess=true` 能启动对应 Flowable 定义。
- 状态映射校验：流程发起状态为 `IN_REVIEW`；一级审批后按配置变为 `LEVEL1_APPROVED`；二级审批和流程结束后保留最终连线显式配置的 `FINAL_SPECIAL`。
- 待办与历史校验：两个审批账号只在各自节点取得待办；流程历史记录两个节点、审批账号、操作文本和审批意见。
- 构建要求：后端验收前使用 `mvn clean package -DskipTests`，避免 IDE 增量编译遗留的错误 class 被普通增量打包带入可执行 JAR。

## 已发现并修复的问题

- 列表多选字段当后端返回 JSON 字符串或逗号分隔字符串时，页面直接显示原始值，现统一解析后按选项 label 显示。
- 日期字段遇到非法日期值时会显示 `Invalid Date`，现显示为 `-`。
- 多运行态表单配置合并时只合并字段、丢失后续表单按钮，现合并并按按钮 key/code/id 去重。
- 按钮权限工具只能依赖已安装的 Pinia Store，不便于独立测试，现支持显式传入权限集合并保持原调用兼容。
- 列表配置设计页字段配置表的拖拽列缺少语义标题，现补充 `排序` 列标题，避免配置审计和可访问性缺口。
- 动态实体列表将分页对象响应当数组处理，真实后端返回 `{ list, total }` 等结构时会导致 `slice` 报错，现兼容数组、`list`、`records`、`rows`。
- 请求模块直接读取 `import.meta.env.VITE_API_BASE_URL`，在 Node/SSR/集成测试运行时会因 `import.meta.env` 不存在而报错，现增加安全访问。
- 流程发布保存状态映射时，旧数据库必填列 `entity_status` 未赋值导致事务回滚，现兼容写入 `entityStatusCode`。
- 动态实体流程状态监听器原先固定更新旧 `entity_data` 表，现按 `entityCode` 解析并更新对应动态实体表。
- 状态监听器原先对所有任务实体事件执行更新，后续任务创建时会提前应用最终状态，现仅处理 `TASK_COMPLETED`。
- 流程历史接口只读取 Flowable Comment，审批意见实际保存在本地待办时会丢失，现回退读取 `process_task`。
- 流程进度页原先只展示内部变量标签，现直接展示处理结果和审批意见，并过滤系统变量。
- 实体发布后再绑定流程时，发布快照没有同步流程定义，发起时报“实体发布快照未绑定流程定义”，现同步最新实体发布快照。
- 无命名空间前缀的 BPMN `<process>` 无法被发布清理器替换流程 key，导致发布成功但运行时找不到定义，现兼容默认命名空间和带前缀 XML。
- 长实体编码创建默认编号规则时，完整编码超过 `entity_code_rule.prefix` 的 20 字符限制，现使用可读前缀加稳定校验尾码，并对手工前缀做长度校验。
- 最终任务状态映射会先写入自定义完成状态，随后被流程结束监听器覆盖为完成分类第一项，现当当前状态已属于结束分类时保留显式映射状态。
- 数据库字段超长异常曾因 SQL 文本包含 `entity_code` 被误报为“实体编码已存在”，现返回准确的字段内容过长提示。

## 当前限制

- 当前环境没有 Playwright/Puppeteer 等浏览器自动化依赖，且网络受限；本轮使用系统 Google Chrome Headless + CDP 覆盖 mock E2E 与真实后端截图验收。后续如允许安装 Playwright，可继续扩展更细的拖拽、上传、弹窗表单填写等点击级交互断言。
- 生产级独立代码审查仍发现服务端 RBAC、任务越权、实例详情 IDOR、终止一致性、并发双启动和发布快照版本漂移等高风险问题；这些不影响本次 happy-path 配置闭环通过，但必须在生产上线前单独整改和补负向测试。
