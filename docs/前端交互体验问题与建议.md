# 前端交互体验问题与建议（实体管理 / 流程管理）

> 审查范围：`workflow-web` 前端的实体管理与流程管理相关页面，重点从 **客户交互易用性、界面显示合理性、是否会给用户造成困扰** 三个角度评估。
> 本文档只提问题与建议，不做任何代码修改。所有问题均有代码位置佐证（`文件:行号`）。
> 审查时间：2026-07-23。

## 四、跨模块共性问题（建议优先统一治理）

1. **未保存数据保护缺失**：实体设计器、表单设计器、流程设计器三个"重编辑器"均无 `onBeforeRouteLeave`/`beforeunload` 守卫，且弹窗普遍允许点遮罩关闭丢输入。建议封装统一的脏数据守卫方案和弹窗防误关规范。
2. **错误处理两极分化**：要么静默吞错（只 `console.error`，界面像"没数据"），要么把后端原始异常/笼统"操作失败"抛给用户。建议统一封装请求错误提示：用户可读文案 + 原始错误进 console + 加载失败给空态和重试。
3. **保存语义混乱**：假保存（FormDesign）、误导性"已保存"提示、部分操作立即持久化部分需手动保存，并存于同一系统。建议统一草稿/发布模型并在 UI 明确标注"未保存/未发布"状态。
4. **危险操作确认不足**：删除字段、删除节点、删除权限规则、删除已发布流程/实体均确认过弱或无确认。建议建立危险操作分级确认规范（普通确认 → 带影响说明 → 输入名称确认）。
5. **技术概念直接暴露给业务用户**：DDL SQL、迁移标记、内容哈希、listKey、DemoJavaDelegate、localhost 示例、英文枚举原值（PENDING/TERMINATED/ACTIVE）等。建议梳理一份"业务用户可见文案"清单，技术信息折叠或仅管理员可见。
6. **新旧页面并存造成功能割裂**：数据管理（EntityDataManage vs EntityDataList）、表单管理（EntityFormManage vs EntityFormList）、流程进度（ProcessDetail vs ProcessProgress）均有两套实现且能力/显示不一致。建议尽快收敛并下线旧页面。
