## 🔴 问题 1：`approved` 变量类型变更导致旧流程条件失效

### 位置
- `workflow-server/src/main/java/com/workflow/service/impl/TaskServiceImpl.java:73`、`:236`、`:258`、`:308`、`:403`、`:819`
- `workflow-server/src/main/java/com/workflow/listener/AutoCompleteTaskListener.java:29`
- `workflow-server/src/main/java/com/workflow/service/TaskActionService.java:206`、`:236`（同样是字符串）
- `workflow-server/src/main/java/com/workflow/service/ProcessRollbackService.java:93`、`:178`

### 问题描述
本次改动把 `approved` 流程变量从布尔值 `true/false` 统一改成了字符串 `"approve"/"reject"`。例如 `TaskServiceImpl.java:236`：
```java
runtimeService.setVariable(processInstanceId, "approved", "approve");  // 原来是 true
```
前端条件表达式编辑器 `NodeConfigPanel.vue:2649` 生成的是字符串比较：
```js
if (fieldType === 'string' || fieldType === 'select') {
  if (value && !value.startsWith('${') && !value.startsWith('#{')) {
    value = `'${value}'`  // approved 会生成 approved == 'approve'
  }
}
```

### 触发场景（配置不生效）
**已经发布上线、且 BPMN XML 里写的是 `${approved == true}` 的旧流程**：
- 用户点"通过" → 后端写入 `approved = "approve"`（字符串）
- 网关条件 `${approved == true}` 判断字符串 `"approve"` 是否等于布尔 `true` → **永远为 false**
- 结果：明明审批通过了，流程却走进了"驳回/拒绝"分支，或者卡在网关不走。

### 为什么"配置了无法使用"
用户在旧流程里配的网关条件，运行时类型对不上，条件永远不成立。新流程用字符串 `'approve'` 才能匹配。

### 建议
- 对历史流程做数据迁移，把 BPMN XML 中 `approved == true` 改为 `approved == 'approve'`；
- 或在 `ProcessBpmnPublishSanitizer` 发布时统一改写条件表达式；
- 网关条件编辑器里给 `approved` 字段加迁移提示。

## 🔴 问题 2：数据权限委托 SQL 字段名硬编码错误

### 位置
- `workflow-server/src/main/java/com/workflow/service/permission/DataPermissionEngine.java:33`
```java
private static final String DELEGATE_USER_FIELD = "create_by";  // 少了一个 d
```
- 对比：`workflow-server/src/main/java/com/workflow/entity/EntityData.java:152`
```java
@TableField("created_by")  // 数据库实际字段
private String createdBy;
```
- 对比：`workflow-server/src/main/resources/db/migration/V001__business_schema.sql` 中所有业务表均使用 `created_by`。

### 问题描述
`DataPermissionEngine.applyDelegation()`（第 265–298 行）在拼装委托 SQL 时用的是 `create_by`（少一个 d）：
```java
String delegateSql = DELEGATE_USER_FIELD + " IN ('" + escapedIds + "')";  // create_by IN (...)
```
而：
- 实际表字段是 `created_by`；
- `FilterConfigDTO.FieldMappingDTO.userField` 默认值也是 `created_by`（第 41 行）；
- `PermissionSqlBuilder.java:35` 的 fallback 也是 `"create_by"`（同样是错的，见问题 7）；
- 前端 `EntityDesign.vue:772`、`:1143` 默认值 `created_by`。

### 触发场景（配置不生效）
用户配置了"数据权限委托"（A 委托给 B），B 登录查看数据时：
- 主规则 SQL 生成 `created_by IN ('A的ID')`（正确字段）
- 委托扩展 SQL 生成 `create_by IN ('A的ID')`（错误字段）
- 合并后：`(created_by IN (...)) OR (create_by IN (...))`
- 数据库执行时 `create_by` 列不存在 → **直接抛 SQL 异常，B 看不到任何数据**，或者（若被全局 catch）返回空列表。

### 为什么"配置了无法使用"
委托配置界面填了委托关系，运行时 SQL 拼出非法字段名，整个列表查询失败。

### 建议
把 `DataPermissionEngine.java:33` 改为 `created_by`，并和 `FilterConfigDTO.userField` 共用同一个常量，避免再次漂移。

## 🔴 问题 3：`isFirstUserTask` BFS 逻辑缺陷，跳过开关永久置灰

### 位置
- `workflow-web/src/components/NodeConfigPanel.vue:1411–1450`

### 问题描述
判断"当前节点是否是第一个用户任务节点"的 BFS 遍历，在遇到非 UserTask 目标节点时**没有把它入队继续遍历**：
```js
for (const connection of outgoing) {
  const target = connection.target
  if (target.type === 'bpmn:UserTask') {
    firstUserTask = target
    break
  }
  if (!visited.has(target.id)) {
    queue.push(target)   // ← 只在"非UserTask"时入队
  }
}
```
看起来没问题，但 `if (!visited.has(target.id))` 的判断在 `UserTask` 分支 break 之后不会执行；真正的问题是：**当 startEvent 的出线直接连到一个排他网关/并行网关/服务任务时**，target 是网关，会入队继续 BFS——这本身对。但 `queue.shift()` 取出的网关，其 `outgoing` 里的 target 如果又是网关，仍能继续。

实际缺陷在于：**BFS 只沿着 `outgoing` 走，没有处理"网关的 default flow"以及"条件为 false 的分支"**。当排他网关有多条出线、且只有默认流能到达第一个 UserTask 时，BFS 会把所有分支都遍历，**可能找到一个并非"运行时真正第一个"的 UserTask**，但更常见的失效是：

更直接的失效场景：`while (queue.length > 0 && !firstUserTask)` 在找到第一个 UserTask 后立即停。如果流程是 `Start → 网关A → (分支1) UserTask_审批 → (分支2) UserTask_归档`，BFS 会把两个 UserTask 都当作"第一个"（取决于 outgoing 顺序），导致**非首节点也被判定为首节点**，从而允许在非首节点上设置跳过——但后端 `WorkflowAutoSkipService` 只按 `skipNode=true` 跳过，结果**跳过了不该跳的节点**。

### 触发场景（配置不生效 / 配置了乱用）
- 流程 `开始 → 排他网关 → 经理审批（分支1）/ 财务审批（分支2）`
- BFS 可能判定"财务审批"也是 firstUserTask（若 outgoing 顺序里它在前面）
- 用户在"财务审批"上开启跳过 → 流程跑到这里被强行跳过 → **财务审批永远不执行**

### 为什么"配置了无法使用"
开关在错误的节点上可用，在真正的首节点上反而置灰（当 outgoing 顺序把非首节点排在前面时）。

### 建议
- BFS 时区分"可达"与"必然可达"；
- 或干脆放宽限制：允许任意用户任务设置跳过，由后端按运行时实际第一个任务判断。

## 🟠 问题 4：`skipNodeEnabled` 变量被移除，后续任务不再跳过

### 位置
- `workflow-server/src/main/java/com/workflow/service/TaskDetailService.java:290`
```java
variables.remove("skipNodeEnabled");
```
- 设置处：`workflow-server/src/main/java/com/workflow/service/EntityDataService.java:171`、`workflow-server/src/main/java/com/workflow/entity/runtime/EntityWorkflowRuntimeService.java:106`
- BPMN 注入：`workflow-server/src/main/java/com/workflow/process/definition/ProcessBpmnPublishSanitizer.java:276` 注入 `${skipNodeEnabled}` 到 `flowable:skipExpression`

### 问题描述
流程启动时设置 `skipNodeEnabled = true`，BPMN 里跳过节点的 `skipExpression="${skipNodeEnabled}"`。但 `TaskDetailService.getTaskDetail()` 在回退取流程变量当表单数据时，**直接从流程实例上 remove 了这个变量**（不是从拷贝里 remove，是从 `runtimeService.getVariables()` 返回的 map 里 remove——虽然 getVariables 返回的是副本，但同方法后续若再 set 回去就会丢）。

更关键的是：这段逻辑只在"实体数据为空、回退用流程变量当表单数据"时触发。一旦走到这个分支，`skipNodeEnabled` 被移除，**如果后续代码把它 set 回流程实例（本段代码没有，但其他路径可能）**，跳过就失效。

但即便不 set 回去，`getVariables` 返回的是快照副本，`remove` 只影响这次返回的 map，不影响流程实例本身。所以**真正的风险在于**：这段代码的意图是"过滤掉系统变量不返回给前端"，但 `skipNodeEnabled` 是运行时控制变量，不应该出现在"表单数据"里——说明**它在流程变量里和业务数据混在了一起**，设计上耦合了。

### 触发场景（配置不生效）
- 节点配置了跳过，但流程变量里 `skipNodeEnabled` 被业务逻辑意外覆盖或清除（例如前端把流程变量当表单回填后又整体 setVariables）
- 跳过表达式 `${skipNodeEnabled}` 求值为 null/false → **节点不跳过，停在跳过节点上等人工处理**

### 为什么"配置了无法使用"
跳过配置依赖一个流程变量，而这个变量在多处被读写/移除，任何一个环节出错都导致 skipExpression 失效。虽然有 `WorkflowAutoSkipService` 兜底（在流程启动时一次性跳过），但**只在启动时跳一次**，流程中途到达的跳过节点不会被兜底处理。

### 建议
- 不要用流程变量承载 `skipNodeEnabled`，改为在 BPMN 里写死 `${true}` 或用专门的 `TaskListener`；
- 或在 `WorkflowAutoSkipService` 里监听任务创建事件（而不仅在启动时遍历）。

## 🟠 问题 5：`PermissionRuleMatcher` 不处理 `EXPRESSION` 匹配条件

### 位置
- `workflow-server/src/main/java/com/workflow/service/permission/PermissionRuleMatcher.java:56–57`
```java
case "EXPRESSION":
    return false;   // ← 表达式类型永远不匹配
```

### 问题描述
权限规则的"匹配条件"支持 `USER/ROLE/DEPT/ALL_USERS/EXPRESSION` 五种范围（见 `MatchConfigDTO.java:25` 注释）。但 `matchesCondition` 对 `EXPRESSION` 直接返回 `false`，意味着**任何配置了表达式匹配的规则永远不命中**。

### 触发场景（配置不生效）
- 用户在权限规则编辑器（`EntityDesign.vue:572`）选了"表达式"范围，填了 Groovy 表达式

### 为什么"配置了无法使用"
UI 提供了表达式输入框（`EntityDesign.vue:572`），但后端匹配器是空实现。

### 建议

## 🟠 问题 6：多实例 `candidateGroups` 被覆盖，回显丢失

### 位置
- `workflow-web/src/components/NodeConfigPanel.vue:2076–2077`
```js
const candidateGroups = rawCandidateGroups ||
    ((assigneeConfig.assigneeType === 'group' || assigneeConfig.assigneeType === 'role') ? assigneeConfig.assigneeValue : '') || ''
```
- 覆盖处：`NodeConfigPanel.vue:2436`
```js
modeling.updateProperties(toRaw(props.element), {
  loopCharacteristics: loop,
  assignee: '${' + (elementVariable || 'assignee') + '}',
  candidateGroups: undefined,   // ← 多实例开启时清空 candidateGroups
  candidateUsers: undefined
})
```

### 问题描述
开启多实例时，`updateMultiInstance()` 把 `candidateGroups` 设为 `undefined`（合理，避免会签所有人看到所有任务）。但保存到扩展属性 `assigneeConfig` 时，`buildAssigneeConfig` 拿的是当前 `assigneeForm.value`——此时 `candidateGroups` 仍保留着用户选的值（因为 `updateMultiInstance` 没清 form 里的字段，只清了 BPMN 属性）。所以扩展属性里有，BPMN 属性里没有。

回显时（第 2076 行）：`rawCandidateGroups` 来自 BPMN（为空），于是 fallback 到 `assigneeConfig.assigneeValue`。但 `buildAssigneeConfig`（在 `@/shared/process-config`）是否一定把 candidateGroups 存到 `assigneeValue`？如果没存，回显就空。

### 触发场景（配置不生效）
- 用户在"用户组"模式下选了组 G1，保存
- 切到"启用多实例"，保存（BPMN candidateGroups 被清空，但扩展属性 assigneeConfig 可能还留着）
- 再切回"关闭多实例"：`onMultiInstanceChange(false)` 调 `updateProperty('candidateGroups', assigneeForm.value.candidateGroups)`——如果这一步 form 里还有 G1，能恢复；但若中间切换过 assigneeType（`onAssigneeTypeChange` 会清空 form 字段），G1 就永久丢失了
- 结果：**重新打开节点配置，用户组选择是空的，但 BPMN 里也没有，执行人配置彻底丢失**

### 为什么"配置了无法使用"
多实例与普通模式来回切换时，执行人配置会丢失。

### 建议
- 关闭多实例时，从 `assigneeConfig` 扩展属性恢复 `candidateGroups`，而不是依赖 form 内存；
- 或 `updateMultiInstance` 不要清空 form 的 candidateGroups，只清 BPMN 属性。

## 🟠 问题 7：`PermissionSqlBuilder` 的 `userField` fallback 字段名错误

### 位置
- `workflow-server/src/main/java/com/workflow/service/permission/PermissionSqlBuilder.java:35`
```java
String userField = safeField(mapping.getUserField(), "create_by");  // 少 d
```
- 对比 `FilterConfigDTO.java:41`：`private String userField = "created_by";`（正确）
- 对比前端 `EntityDesign.vue:772`：`fieldMapping: { userField: 'created_by', ... }`（正确）

### 问题描述
`safeField(fieldName, fallback)` 当 `fieldName` 为空时用 fallback。这里 fallback 是 `create_by`（错），但 DTO 默认值是 `created_by`（对）。**只有在 `fieldMapping` 整个为 null、或 `userField` 显式传空字符串时**才会走到 fallback。

`DataPermissionEngine.doCalculatePermission` 第 203 行 `parseFilterConfig` 解析 JSON 到 DTO，Jackson 会使用 DTO 默认值 `created_by`，所以正常路径不会触发 fallback。但**如果 filterConfig JSON 里显式写了 `"userField":""` 或 `"userField":null`**，Jackson 会覆盖默认值为 null/空，于是走到 fallback `create_by`——错误字段。

### 触发场景（配置不生效）
- 用户在字段映射里把"用户字段"清空（或保存过空值）
- 运行时 `safeField(null, "create_by")` → `create_by`
- SQL 生成 `create_by = '用户ID'` → 字段不存在 → 查询失败或返回空

### 为什么"配置了无法使用"
字段映射留空时，fallback 字段名与数据库不一致。

### 建议
- `PermissionSqlBuilder.java:35` fallback 改为 `created_by`；
- 与 `FilterConfigDTO` 共用常量；
- 同步检查 `DataPermissionEngine.java:33` 的 `DELEGATE_USER_FIELD`（问题 2）。

## 🟡 问题 8：条件表达式切换"无条件"后回显串味

### 位置
- `workflow-web/src/components/NodeConfigPanel.vue:2236–2244`

conditionForm.value = {

### 建议
`onConditionTypeChange('')` 里同步 `conditionList.value = [{ property:'', operator:'==', value:'', logic:'&&' }]`。

## 🟡 问题 9：权限规则更新丢失 `createdBy`

### 位置

## 🟡 问题 10：`WorkflowAutoSkipService` 最多迭代 10 次

### 位置
- `workflow-server/src/main/java/com/workflow/service/WorkflowAutoSkipService.java:57`
```java
int maxIterations = 10;
for (int i = 0; i < maxIterations; i++) { ... }
```

### 问题描述
自动跳过用循环 + `completedTaskIds` 去重来逐个完成跳过节点。每次循环查一次 active task，跳过其中的 skip 节点。最多 10 次。如果流程有超过 10 个连续的跳过节点（或跳过节点之间夹着会签多实例导致一次循环只跳一个），**第 11 个之后的跳过节点不会被跳过**。

### 触发场景
长流程链路里大量节点配置了跳过（如迁移历史流程时批量设置），第 11 个跳过节点开始失效，流程停在那里等人工。

## 附：其他值得关注的配置链路风险（未深挖）

- `TaskActionService.java:191` 判断多实例用的是 `vars.containsKey("nrOfInstances")`，但 `nrOfInstances` 是多实例**内部**变量，在 `getVariables(processInstanceId)`（实例级）可能取不到，导致会签驳回的"一人驳回即整体驳回"逻辑失效。
